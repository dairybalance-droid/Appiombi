# Appiombi Data Model

## Principles

- English names for tables and fields.
- UUID primary keys.
- Flat, explicit foreign keys for FlutterFlow compatibility.
- Clear separation between auth/access, farm domain data, legal consent data, audit, and SaaS subscription data.
- Ready for offline sync and manual conflict review.
- Privacy by design: collect only needed business data and keep access bounded by `farm_id` and role.

## Main Identity And Access Entities

### auth.users

Managed by Supabase Auth.

Responsibilities:

- email and password authentication
- email verification
- password reset
- session issuance and refresh token lifecycle

### profiles

Application profile linked 1:1 to `auth.users`.

Key fields:

- `id`
- `auth_user_id`
- `email`
- `first_name`
- `last_name`
- `display_name`
- `default_role`
- `account_status`
- `is_active`
- `email_verified_at`
- `must_accept_legal_documents`
- `marketing_consent`
- `mfa_enrolled`
- `mfa_required`

Purpose:

- stores product-level identity and security posture
- supports role-aware flows
- avoids putting secrets into app tables

### farm_users

Grants access to a farm.

Key fields:

- `id`
- `farm_id`
- `profile_id`
- `role`
- `status`
- `granted_by_profile_id`
- `accepted_invite_id`
- `granted_at`
- `revoked_at`

Rules:

- Farmer ownership is represented by `farms.owner_profile_id`.
- Shared access for veterinarians and trimmers is represented here.
- Future `farm_collaborator` access also belongs here.
- Revocation must immediately stop backend access through RLS.
- UI and standard operational queries should prefer `active_farm_users` when the app needs non-deleted active memberships only.

### farm_invites

Invite codes generated by the farmer.

Key fields:

- `id`
- `farm_id`
- `invited_email`
- `invite_code`
- `invited_role`
- `created_by_profile_id`
- `expires_at`
- `accepted_at`
- `revoked_at`
- `status`

Rules:

- Email can be nullable to allow generic code-based sharing.
- Invite code remains unique.
- Accepted or revoked invites must not be reusable.

## Legal And Consent Entities

### legal_documents

Key fields:

- `id`
- `document_type`
- `version`
- `locale`
- `content_url`
- `effective_date`
- `is_active`

Purpose:

- tracks which legal text version is active
- allows re-consent when mandatory versions change

### user_consents

Traceable user consent records.

Key fields:

- `id`
- `user_id`
- `legal_document_id`
- `accepted_at`
- `ip_address`
- `user_agent`
- `consent_type`
- `is_required`
- `revoked_at`

Rules:

- each acceptance is stored as an auditable event
- marketing consent remains separate and optional
- required privacy and terms consents must match the active versions

## SaaS And Account State Entities

### subscriptions

Farm-level SaaS subscription state.

Key fields:

- `id`
- `farm_id`
- `owner_user_id`
- `provider`
- `provider_customer_id`
- `provider_subscription_id`
- `plan`
- `status`
- `current_period_start`
- `current_period_end`
- `grace_period_until`

Rules:

- one farm may have multiple historical subscriptions
- one active/current subscription state governs write access
- expired access must not auto-delete farm data

### audit_logs

Key fields:

- `id`
- `actor_user_id`
- `farm_id`
- `entity_type`
- `entity_id`
- `action`
- `old_values`
- `new_values`
- `created_at`
- `ip_address`
- `device_id`

Purpose:

- security traceability
- access accountability
- change review for regulated business operations

## Farm Domain Entities

### farms

Represents a dairy business workspace.

Key fields:

- `id`
- `owner_profile_id`
- `name`
- `farm_code`
- `street_address`
- `street_number`
- `postal_code`
- `city`
- `province`
- `region`
- `country`
- `is_active`

Rules:

- `farms.id` is the only guaranteed unique identifier for farm identity.
- Farm name must not be used as unique identity in app logic.
- Different farms may share the same name and even the same street name.
- UI should use address fields to disambiguate visually similar farms.

### cows

Farm-specific cow registry.

Key fields:

- `id`
- `farm_id`
- `cow_identifier`
- `display_identifier`
- `is_active`

Rules:

- unique on `(farm_id, cow_identifier)`
- same identifier allowed in different farms
- UI and standard operational queries should prefer `active_animals`.

### trimming_sessions

Top-level session container.

Key fields:

- `id`
- `farm_id`
- `created_by_profile_id`
- `started_at`
- `ended_at`
- `status`
- `title`
- `notes`

Rules:

- session may span multiple days
- session may be reopened and edited
- UI and standard operational queries should prefer `active_trimming_sessions`.

### trimming_session_days

Optional explicit day log for non-consecutive session work.

### cow_visits

One saved visit for one cow inside one session.

Key fields:

- `id`
- `session_id`
- `farm_id`
- `cow_id`
- `visit_date`
- `insertion_index`
- `criticality_score`
- `sole_count`
- `bandage_count`
- `corkscrew_grade`
- `laminitis_status`
- `antibiotic_given`
- `antiinflammatory_given`
- `straw_box_required`
- `evaluate_culling`
- `is_chronic_cow`
- `other_flag`
- `notes`

Rules:

- unique on `(session_id, cow_id)`
- prevents duplicate same-cow save in one session
- UI and standard operational queries should prefer `active_cow_visits`.

### claw_zone_observations

Normalized lesion records for horn and derm areas.

UI and standard operational queries should prefer `active_claw_observations`.

### cow_visit_flags

Optional extensibility table for future non-core observations.

Examples:

- future lesion photos metadata
- future BCS
- future tit scoring
- future non-podal lesions

### tasks

Operational follow-up tasks.

UI and standard operational queries should prefer `active_clinical_tasks`.

### task_deliveries

Predisposizione futura delivery tracking for in-app, email, or WhatsApp channel state.

## Active Views Strategy

Soft-deletable operational tables should expose read-safe views for normal app usage:

- `active_farm_users`
- `active_animals`
- `active_trimming_sessions`
- `active_cow_visits`
- `active_claw_observations`
- `active_clinical_tasks`

These views are intended for standard UI and app queries so `deleted_at` rows are excluded by default.

## Relationship Summary

- one auth user maps to one profile
- one profile owns zero or more farms
- one farm has many farm users
- one farm has many cows
- one farm has many sessions
- one farm has many legal and operational audit events
- one farm may have many historical subscriptions
- one user may accept many legal document versions over time
- one session has many session days
- one session has many cow visits
- one cow has many cow visits across sessions in the same farm
- one cow visit has many claw zone observations
- one cow visit may generate many tasks

## Access Model Notes

- `can_access_farm(farm_id)` controls read access at farm boundary.
- `can_write_farm(farm_id)` controls write access and can enforce subscription read-only behavior.
- `farm_access_modes` can expose readable per-farm access state without exposing raw subscription rows to invited operators.
- `super_admin` is internal and should be granted through explicit profile role, not through client secrets.

## Farm List Display Notes

When listing farms in the UI, show:

- farm name
- full street address
- city
- province
- optional `farm_code`

Example:

- Azienda Agricola Rossi
  Via Roma 12, 42025 Cavriago (RE)
- Azienda Agricola Rossi
  Via Roma 14, 42025 Cavriago (RE)

## FlutterFlow Compatibility Notes

- prefer simple enums and booleans over nested JSON for core fields
- keep lesion data relational, but small enough for filtered queries
- include direct `farm_id` on `cow_visits`, `tasks`, and sync tables for simpler query building
- use views or RPCs later for login gating, legal checks, and dashboard aggregation
